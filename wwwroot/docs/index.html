Email / Password Flow

Registration and login with BCrypt hashing. OAuth providers are dynamically detected.

Register — AuthService.cs
public async Task<AuthResult> RegisterAsync(RegisterDto dto)
{
    if (await _db.Users.AnyAsync(u => u.Email == dto.Email))
        throw new InvalidOperationException("Email already registered");

    var user = new User
    {
        Username     = dto.Username,
        Email        = dto.Email,
        PasswordHash = BCrypt.Net.BCrypt.HashPassword(dto.Password),
        AuthProvider = "local"
    };

    _db.Users.Add(user);
    await _db.SaveChangesAsync();
    return await GenerateTokensAsync(user);
}
Login — AuthService.cs
public async Task<AuthResult> LoginAsync(LoginDto dto)
{
    var user = await _db.Users.FirstOrDefaultAsync(u =>
        u.Email == dto.Email && u.AuthProvider == "local");

    if (user == null || !BCrypt.Net.BCrypt.Verify(dto.Password, user.PasswordHash))
        throw new UnauthorizedAccessException("Invalid credentials");

    return await GenerateTokensAsync(user);
}

OAuth SSO Flow

Google/GitHub OAuth2 with automatic account linking when emails match.

LoginWithExternalProviderAsync — AuthService.cs
public async Task<AuthResult> LoginWithExternalProviderAsync(
    string provider, string externalUserId, string email, string username)
{
    // 1. Find by provider + external ID
    var user = await _db.Users.FirstOrDefaultAsync(u =>
        u.AuthProvider == provider && u.ExternalUserId == externalUserId);

    if (user == null)
    {
        // 2. Try linking to existing email account
        user = await _db.Users.FirstOrDefaultAsync(u => u.Email == email);
        if (user != null)
        {
            user.AuthProvider   = provider;
            user.ExternalUserId = externalUserId;
        }
        else
        {
            // 3. Create new SSO-only user (no password)
            user = new User
            {
                Username       = username,
                Email          = email,
                AuthProvider   = provider,
                ExternalUserId = externalUserId,
                EmailConfirmed = true
            };
            _db.Users.Add(user);
        }
    }

    await _db.SaveChangesAsync();
    return await GenerateTokensAsync(user);
}

OAuth Sequence

1. Frontend → GET /api/auth/google/login

2. Server → 302 Redirect to Google consent screen

3. User approves → Google redirects back with auth code

4. Server → Exchange code for tokens at Google

5. Server → GET userinfo (email, name, sub)

6. Server → Find or create User in DB

7. Server → Set HttpOnly cookie + Return { accessToken, user }

Token Refresh Flow

Single-use refresh token rotation. Each refresh revokes the old token and issues a new pair.

Why Do We Need Refresh Tokens?

The Problem: Access tokens are short-lived (15 min) for security. But forcing users to re-login every 15 minutes is terrible UX.

The Solution: Refresh tokens are long-lived (7 days) and stored securely. They let the server issue new access tokens without re-authentication.

Complete Flow Diagram

Step-by-Step Breakdown

1

Access Token Expires

After 15 minutes, the JWT access token expires. Any API request with an expired token returns 401 Unauthorized.

GET /api/questions → 401 { "error": "Token expired" }

2

Frontend Interceptor Catches 401

The Axios interceptor automatically detects the 401 error and attempts to refresh the token before the user notices anything.

if (error.response?.status === 401 && !req._retry) {

req._retry = true;

// attempt refresh...

}

3

Browser Sends HttpOnly Cookie Automatically

Frontend calls POST /api/auth/refresh. Browser automatically includes the HttpOnly cookie - JavaScript cannot access or modify it.

// Frontend code (no refresh token handling!)

await axios.post("/api/auth/refresh", null, {

withCredentials: true

});

// Browser automatically sends: Cookie: refresh_token=xxx

4

Server Validates Token

Server checks the refresh token against the database with three validations:

EXISTS?

Token found in DB

EXPIRED?

ExpiresAt > UtcNow

REVOKED?

IsRevoked == false

5

Token Rotation (If Valid)

If all checks pass, server immediately revokes the old refresh token and issues a new pair. This is "rotation" - each token can only be used once.

// Revoke old token (single-use enforcement)

storedToken.IsRevoked = true;

// Issue new pair

return await GenerateTokensAsync(storedToken.User);

6

New Tokens Returned

Server returns only access token in body. Refresh token is set as HttpOnly cookie automatically by browser.

// Response body (only access token)

{

"accessToken": "eyJhbGciOiJI...",

"user": { "id": 1, "email": "..." }

}

// Refresh token is in HttpOnly cookie (JS cannot read)

Set-Cookie: refresh_token=xxx; HttpOnly; Secure; SameSite=Strict

Invalid Token Scenarios

Scenario A: Token Expired

Refresh token exceeded 7-day limit. User must re-login.

Server: storedToken.ExpiresAt < DateTime.UtcNow

→ 401 "Refresh token expired. Please login again."

Scenario B: Token Revoked (Theft Detection)

Token was already used (possibly by attacker). Server detects replay.

Server: storedToken == null (revoked tokens filtered out)

→ 401 "Invalid refresh token. Possible theft detected."

RefreshTokensAsync — AuthService.cs (HttpOnly Cookie)
public async Task<AuthResult> RefreshTokensAsync(HttpContext context)
{
    // 1. Get refresh token from HttpOnly cookie (not from request body!)
    var refreshToken = context.Request.Cookies["refresh_token"];
    if (string.IsNullOrEmpty(refreshToken))
        throw new UnauthorizedAccessException("Refresh token not found");

    // 2. Find token in DB, include User
    var storedToken = await _db.RefreshTokens
        .Include(t => t.User)
        .FirstOrDefaultAsync(t =>
            t.Token == refreshToken && !t.IsRevoked);

    // 3. Validate
    if (storedToken == null || storedToken.ExpiresAt < DateTime.UtcNow)
        throw new UnauthorizedAccessException("Invalid refresh token");

    // 4. Revoke old token (single-use rotation)
    storedToken.IsRevoked = true;

    // 5. Issue new pair (sets new HttpOnly cookie automatically)
    return await GenerateTokensAsync(storedToken.User, context);
}

private async Task<AuthResult> GenerateTokensAsync(User user, HttpContext context)
{
    var accessToken = _tokenService.GenerateAccessToken(user);
    var refreshToken = _tokenService.GenerateRefreshToken();

    // Store in DB
    _db.RefreshTokens.Add(new RefreshToken
    {
        Token = refreshToken.Token,
        ExpiresAt = refreshToken.ExpiresAt,
        UserId = user.Id
    });
    await _db.SaveChangesAsync();

    // Set HttpOnly cookie - JavaScript CANNOT access this!
    var cookieOptions = new CookieOptions
    {
        HttpOnly = true,      // JS cannot read
        Secure = true,        // HTTPS only
        SameSite = SameSiteMode.Strict,  // CSRF protection
        Path = "/api/auth",   // Only sent to auth endpoints
        Expires = refreshToken.ExpiresAt
    };
    context.Response.Cookies.Append("refresh_token", refreshToken.Token, cookieOptions);

    // Only return access token in body (short-lived, less risk)
    return new AuthResult
    {
        AccessToken = accessToken,
        User = new UserDto { Id = user.Id, Username = user.Username, Email = user.Email }
    };
}

Security Deep Dive (Maximum Protection)

HttpOnly Cookie (XSS Protection)

Refresh token stored in HttpOnly cookie. JavaScript cannot access it via document.cookie. XSS attacks cannot steal the refresh token.

Secure Flag (HTTPS Only)

Cookie only sent over HTTPS. Prevents network sniffing attacks from stealing tokens in transit.

SameSite=Strict (CSRF Protection)

Cookie only sent for same-site requests. Prevents Cross-Site Request Forgery attacks.

Path Restriction

Cookie only sent to /api/auth endpoints. Reduces attack surface.

Single-Use Tokens

Each refresh token can only be used once. After use, IsRevoked = true. If someone steals and uses it, the legitimate user's next refresh attempt will fail → alerting them to compromise.

HSTS (1 Year + Preload)

Forces browsers to use HTTPS for 1 year. Added to HSTS preload list. Prevents downgrade attacks.

XSS Cannot Steal Refresh Token

With HttpOnly cookie, even if XSS injects malicious script, it cannot read document.cookie. Refresh token is completely invisible to JavaScript.

What Happens If a Refresh Token Is Stolen?

XSS ATTACK: BLOCKED

With HttpOnly cookie, XSS cannot steal refresh token. document.cookie returns empty for refresh_token.

NETWORK SNIFFING: BLOCKED

HTTPS encrypts all traffic. Cookie with Secure flag only sent over HTTPS.

CSRF ATTACK: BLOCKED

With SameSite=Strict, cookie only sent for same-site requests. Cross-site requests cannot include it.

STILL PROTECTED: Single-Use Rotation

If somehow stolen (e.g., physical access), single-use rotation ensures: first use = success, second use = 401. Legitimate user is alerted.

MAXIMUM SECURITY ACHIEVED

  • HttpOnly: JS cannot access refresh token
  • Secure: Only sent over HTTPS
  • SameSite=Strict: CSRF protection
  • Path=/api/auth: Limited exposure
  • Single-use: Theft detection

Services

Token generation with HMAC-SHA256 signing and cryptographically secure refresh tokens.

TokenService.cs — key methods
public string GenerateAccessToken(User user)
{
    var key = new SymmetricSecurityKey(Encoding.UTF8.GetBytes(_jwtSettings.SecretKey));
    var claims = new[]
    {
        new Claim(JwtRegisteredClaimNames.Sub, user.Id.ToString()),
        new Claim(JwtRegisteredClaimNames.Email, user.Email),
        new Claim(JwtRegisteredClaimNames.Jti, Guid.NewGuid().ToString()),
        new Claim("auth_provider", user.AuthProvider ?? "local")
    };

    var token = new JwtSecurityToken(
        issuer: _jwtSettings.Issuer,
        audience: _jwtSettings.Audience,
        claims: claims,
        expires: DateTime.UtcNow.AddMinutes(_jwtSettings.AccessTokenExpirationMinutes),
        signingCredentials: new SigningCredentials(key, SecurityAlgorithms.HmacSha256));

    return new JwtSecurityTokenHandler().WriteToken(token);
}

public RefreshTokenResponse GenerateRefreshToken()
{
    var bytes = new byte[64];
    using var rng = RandomNumberGenerator.Create();
    rng.GetBytes(bytes);

    return new RefreshTokenResponse
    {
        Token = Convert.ToBase64String(bytes),
        ExpiresAt = DateTime.UtcNow.AddDays(_jwtSettings.RefreshTokenExpirationDays)
    };
}

Frontend Integration

Axios interceptor with automatic 401 → refresh → retry. Refresh token is handled automatically via HttpOnly cookie.

api.ts — Axios with auto-refresh (HttpOnly Cookie)
// Only access token in memory - refresh token is in HttpOnly cookie!
let accessToken: string | null = null;

const api = axios.create({ 
  baseURL: "/api",
  withCredentials: true  // Important: Send cookies with requests
});

// Add access token to requests
api.interceptors.request.use((config) => {
  if (accessToken) {
    config.headers.Authorization = `Bearer ${accessToken}`;
  }
  return config;
});

// Handle 401 → auto refresh
api.interceptors.response.use(
  (res) => res,
  async (error) => {
    const req = error.config;
    
    // If 401 and not already retried
    if (error.response?.status === 401 && !req._retry) {
      req._retry = true;
      
      try {
        // Just call refresh - browser sends HttpOnly cookie automatically!
        // No need to send refresh token in body
        const { data } = await axios.post("/api/auth/refresh", null, {
          withCredentials: true
        });
        
        // Only access token comes in response body
        accessToken = data.accessToken;
        req.headers.Authorization = `Bearer ${data.accessToken}`;
        
        return api(req);  // Retry original request
      } catch {
        // Refresh failed - redirect to login
        accessToken = null;
        window.location.href = "/login";
      }
    }
    return Promise.reject(error);
  }
);

// Login function
export async function login(email: string, password: string) {
  const { data } = await api.post("/auth/login", { email, password });
  accessToken = data.accessToken;
  // Refresh token is now in HttpOnly cookie - we don't touch it!
  return data.user;
}

// Logout function
export async function logout() {
  await api.post("/auth/logout");
  accessToken = null;
  // HttpOnly cookie is cleared by the server
}

HttpOnly Cookie (Refresh Token)

JavaScript cannot access. XSS attacks cannot steal refresh tokens.

Secure Flag (HTTPS Only)

Cookies only sent over HTTPS. Prevents network sniffing attacks.

SameSite=Strict (CSRF Protection)

Cookies only sent for same-site requests. Blocks CSRF attacks.

15-min Access Tokens

Limits exposure window. Stale tokens are useless quickly.

Single-Use Refresh Rotation

Each refresh revokes old token. Stolen tokens can't be reused.

HSTS (1 Year + Preload)

Forces HTTPS. Prevents downgrade attacks. Added to preload list.

BCrypt (Cost 12)

~250ms per hash. Built-in salt. Resistant to rainbow tables.

Per-IP Rate Limiting

Auth: 10 req/15min. Global: 100 req/min. Blocks brute force.

COOP, COEP, CORP headers. Prevents Spectre-type attacks.

Content Security Policy

Strict CSP. Blocks inline scripts, restricts resource loading.

ClockSkew = Zero

No 5-min default tolerance. Tokens expire exactly when exp says.

Structured Error Responses

No stack traces leaked. Consistent JSON error format with traceId.

Dynamic OAuth Providers

Only registered if configured. App runs without OAuth secrets.

No Server Banner

Removes Server/X-Powered-By headers. Hides tech stack.
